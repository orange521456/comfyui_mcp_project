# 05 - 模板系统设计

## 设计目标

提供预置经典工作流模板 + 支持用户自定义模板 + Pipeline IR 动态构建，三者结合覆盖所有使用场景。

## 三层工作流构建策略

| 层级 | 方式 | 适用场景 | 灵活性 |
|------|------|---------|--------|
| 1. 内置模板 | `load_template("txt2img", params)` | 快速文生图/图生图 | 低（仅参数替换） |
| 2. Pipeline IR | `build_workflow(pipeline_ir)` | 自定义组合（LoRA/ControlNet/Upscale等） | 高（模块自由组合） |
| 3. 原子操作 | `create_node` + `connect_nodes` | 极端复杂/非标准工作流 | 最高（完全手动） |

Agent 应优先使用 Pipeline IR（层级 2），因为它覆盖了绝大多数场景且使用简单。当 Pipeline IR 无法满足时，回退到原子操作（层级 3）。

## 模板存储

### 内置模板

路径：`src/templates/`

| 模板文件 | 说明 | 可参数化字段 |
|----------|------|-------------|
| `txt2img.json` | 标准文生图 | checkpoint, prompt(+/-), width, height, steps, cfg, sampler, scheduler, seed, batch_size |
| `img2img.json` | 图生图 | 同上 + input_image, denoise |
| `inpaint.json` | 局部重绘 | 同上 + input_image, mask_image, grow_mask_by, denoise |
| `upscale.json` | 超分放大 | input_image, upscale_model, filename_prefix |

> 内置模板保留用于快速场景，但不再扩展新类型。新的工作流类型通过 Pipeline IR 实现。

### 用户自定义模板

路径：`user_templates/`

用户可以将 ComfyUI 导出的 `.json` 工作流文件放入此目录，或通过 `save_workflow` 保存 Pipeline IR 构建的工作流。MCP Server 启动时自动扫描并暴露给 Agent。

## 模板参数化机制

### 内置模板参数化

内置模板使用**占位符**标记可替换参数：

```json
{
  "3": {
    "inputs": {
      "seed": "{{seed}}",
      "steps": "{{steps}}",
      "cfg": "{{cfg}}",
      "sampler_name": "{{sampler_name}}",
      "scheduler": "{{scheduler}}",
      "denoise": "{{denoise}}",
      "model": ["1", 0],
      "positive": ["2", 0],
      "negative": ["2", 1],
      "latent_image": ["4", 0]
    },
    "class_type": "KSampler"
  }
}
```

Agent 调用 `load_template("txt2img", params={...})` 时，Server 将占位符替换为实际值。

### 用户模板参数化

用户模板是完整的 ComfyUI JSON，没有占位符。Agent 加载后需要通过**原子操作**（`update_node`）来修改参数。

### Pipeline IR 构建

Pipeline IR 不需要模板文件，Agent 直接描述 pipeline 步骤序列，引擎动态生成完整工作流。详见 [03-工作流中间表示.md](./03-工作流中间表示.md)。

## 模板加载流程

```
load_template("txt2img", {checkpoint: "xxx.safetensors", width: 1024, ...})
        │
        ▼
1. 查找模板文件
   ├── 内置模板 → src/templates/txt2img.json
   └── 用户模板 → user_templates/{name}.json
        │
        ▼
2. 如果是内置模板：
   ├── 读取模板 JSON
   ├── 合并用户提供的 params（覆盖默认值）
   ├── 替换所有 {{占位符}}
   └── 返回完整工作流
        │
3. 如果是用户模板：
   ├── 读取模板 JSON
   ├── 如果有 params，遍历所有节点尝试匹配更新
   └── 返回完整工作流
        │
        ▼
4. 存入当前工作流上下文（内存）
   Agent 可继续使用原子操作修改
```

## Pipeline IR 构建流程

```
build_workflow({type: "pipeline", pipeline: [...], meta: {...}})
        │
        ▼
1. 校验 pipeline 结构
   ├── 检查 type == "pipeline"
   ├── 检查 pipeline 数组非空
   └── 检查每个模块名在注册表中
        │
        ▼
2. 模块引擎处理
   ├── 按顺序遍历 pipeline 中的每个模块
   ├── 创建对应节点（简单模块 1 个节点，复合模块多个节点）
   ├── 自动连线：查找最近的生产者
   └── 维护输出类型注册表
        │
        ▼
3. 存入当前工作流上下文（内存）
   Agent 可继续使用原子操作修改
```
