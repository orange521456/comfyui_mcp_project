# 02 - 架构设计

## 整体架构

```
┌──────────────┐     MCP协议      ┌─────────────────┐    REST API     ┌──────────────┐
│  Agent(Trae) │ ◄──────────────► │  MCP Server      │ ◄─────────────► │  ComfyUI     │
│              │   tools/resources │  (Python)        │  (后期+WS)      │  本地服务     │
└──────────────┘                  │                  │                │              │
                                  │ ┌──────────────┐ │                │              │
                                  │ │ 工具层(Tools) │ │                │              │
                                  │ │ 模板管理      │ │                │              │
                                  │ │ 工作流构建    │ │                │              │
                                  │ │ API客户端     │ │                │              │
                                  │ └──────────────┘ │                └──────────────┘
                                  └─────────────────┘
```

## 三层架构

### 1. API 客户端层 (`comfyui_client.py`)

封装与 ComfyUI 的所有 HTTP 通信：

- 模型列表查询
- 节点类型定义获取
- 工作流提交执行
- 执行状态轮询
- 历史记录获取
- 图片下载
- 图片上传

### 2. 领域逻辑层

| 模块 | 职责 |
|------|------|
| `workflow_builder.py` | 中间表示(IR) → ComfyUI 原生 JSON 的翻译 |
| `templates.py` | 模板加载、参数化、管理 |
| `nodes.py` | 节点类型定义缓存与查询 |

### 3. MCP 工具层 (`tools/`)

将领域能力暴露为 MCP tools，供 Agent 调用。详见 [04-MCP工具设计.md](./04-MCP工具设计.md)。

## 配置项

MCP Server 需要以下配置：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| COMFYUI_BASE_URL | ComfyUI 服务地址 | `http://127.0.0.1:8188` |
| USER_TEMPLATES_DIR | 用户模板目录 | `./user_templates/` |
| POLL_INTERVAL | 状态轮询间隔(秒) | 1 |
| POLL_TIMEOUT | 执行超时时间(秒) | 300 |
| THUMBNAIL_MAX_SIZE | 缩略图最大边长(px) | 512 |
| LOG_LEVEL | 日志级别 | `INFO` |
| LOG_DIR | 日志文件目录 | `./logs/` |
| LOG_MAX_DAYS | 日志保留天数 | 7 |

## ComfyUI 服务自动发现

MCP Server 启动时自动检测 ComfyUI 是否可达：

```
启动流程:
  1. 加载配置 COMFYUI_BASE_URL
  2. 发送 GET {COMFYUI_BASE_URL}/system_stats 检测连通性
  3. 如果可达 → 正常启动，记录 ComfyUI 信息
  4. 如果不可达 → 记录 ERROR 日志，但仍启动 MCP Server
     Agent 调用工具时会收到明确的 "ComfyUI 不可达" 错误
```

- 不阻塞启动：即使 ComfyUI 未运行，MCP Server 也正常启动，Agent 调用工具时再报错
- 后续可扩展：监听 ComfyUI 启动/关闭事件，动态更新连接状态

## 统一错误格式

所有 MCP 工具返回统一的 JSON 格式：

```python
# 成功返回
{
    "success": true,
    # ... 工具特定的返回数据
}

# 失败返回
{
    "success": false,
    "error": "人类可读的错误描述",
    "error_code": "COMFYUI_UNREACHABLE",  # 可选，机器可读的错误码
    "detail": "..."  # 可选，详细错误信息（调试用）
}
```

### 预定义错误码

| 错误码 | 说明 |
|--------|------|
| `COMFYUI_UNREACHABLE` | ComfyUI 服务不可达 |
| `COMFYUI_TIMEOUT` | ComfyUI 请求超时 |
| `COMFYUI_ERROR` | ComfyUI 返回错误 |
| `INVALID_PARAM` | 参数校验失败 |
| `TEMPLATE_NOT_FOUND` | 模板不存在 |
| `WORKFLOW_NOT_FOUND` | 当前没有加载工作流 |
| `EXECUTION_FAILED` | 工作流执行失败 |
| `FILE_NOT_FOUND` | 文件不存在 |
| `INTERNAL_ERROR` | 服务器内部错误 |

## MCP Resources（只读数据）

以下只读数据通过 MCP Resource 机制暴露，Agent 可以订阅和读取：

| Resource URI | 内容 | 说明 |
|--------------|------|------|
| `comfyui://models/checkpoints` | checkpoint 列表 | 主模型 |
| `comfyui://models/loras` | LoRA 列表 | |
| `comfyui://models/vaes` | VAE 列表 | |
| `comfyui://models/upscalers` | upscale 模型列表 | |
| `comfyui://models/controlnet` | ControlNet 模型列表 | |
| `comfyui://node_types` | 所有节点类型定义 | 节点元数据 |
| `comfyui://system_info` | ComfyUI 系统信息 | 版本、Python 版本等 |

> 与 Tool 的区别：Tool 是 Agent 主动调用，Resource 是 Agent 订阅后被动获取更新。模型列表变化频率低，Resource 更合适。

## 日志与调试机制

### 日志输出策略

```
日志输出:
  ├── stderr（控制台）  ← 实时查看，不污染 MCP 协议的 stdio 通道
  └── 文件（logs/）     ← 按天轮转，长期保留

关键原则：日志绝不能走 stdout，会破坏 MCP 的 stdio 通信。
```

### 日志级别

| 级别 | 环境变量 | 说明 | 示例 |
|------|----------|------|------|
| DEBUG | `LOG_LEVEL=DEBUG` | 最详细，包含 ComfyUI 请求/响应详情 | 开发调试时启用 |
| INFO | `LOG_LEVEL=INFO`（默认） | 关键操作节点 | 日常运行 |
| WARNING | `LOG_LEVEL=WARNING` | 需注意但不影响运行 | 生产环境 |
| ERROR | `LOG_LEVEL=ERROR` | 功能受阻 | 仅记录错误 |

### 日志内容

- 所有 MCP 工具的调用和返回值（DEBUG 级别）
- ComfyUI API 请求/响应详情（DEBUG 级别）
- 工作流执行生命周期（提交、进度、完成、失败）
- 异常和错误堆栈

### 文件轮转

- 按天轮转：`mcp_server_2026-06-21.log`
- 保留 7 天，自动清理旧日志
- 单文件无大小限制（日切已足够）
